Privacy

Privacy notice

Effective: May 9, 2026 Version: 1.0 Source: github.com/geemeows/clearday

Summary — what actually happens to your data

  • Devy is self-hosted. Your data lives in your Supabase project, not ours.
  • The auth-proxy we operate is stateless: it verifies an HMAC and 302-redirects. No tokens, codes, emails, or request bodies are stored or logged with identifying content.
  • Devy reads metadata from GitHub, Google Calendar, and Slack on a polling schedule. It does not read message bodies, file contents, or PR diffs.
  • Devy writes to providers only on user-initiated actions — currently a Focus session that creates a Calendar event and sets your Slack status & Do Not Disturb.
  • Raw signal data ages out after 90 days; aggregate counts (pulse rollups) are kept for charts.
  • You can revoke any provider in Settings, delete all your data with one Postgres call, or destroy the whole deployment.

1. Who is the data controller?

You are. Because Devy is single-tenant and self-hosted, the operator of each deployment — the person who provisioned the Supabase project, deployed the Worker, and set ALLOWED_EMAIL — is the data controller for everything that flows through that deployment. The Clearday project does not operate a hosted version of Devy and never holds copies of your data.

The only exception is the auth-proxy described below, where the project is the controller for the limited, transient processing it performs.

2. The auth-proxy

OAuth flows from GitHub, Google and Slack are redirected through a Cloudflare Worker we operate at https://auth.clearday.geemeows.com. Here is the complete list of what the proxy does with your data:

  • It receives the provider redirect (which contains an opaque OAuth code and a signed state parameter).
  • It verifies the state against an HMAC secret and a 10-minute TTL.
  • If valid, it 302-redirects the browser to the userBackendUrl embedded in the state, with the same code attached.

The proxy never exchanges the code for a token, never opens the request body, never persists anything to a database, and never writes the code, your email, or the redirect target to a log. Cloudflare's standard edge metrics (request count, status codes, region, anonymised IP for abuse-prevention windows of up to 24 hours) apply to all Workers and are subject to Cloudflare's privacy policy.

Practical effect: the OAuth code exists in the proxy's memory for a few milliseconds, only long enough to be redirected. The access_token and refresh_token are then exchanged exclusively between your Worker and the provider — they never traverse the proxy.

3. What Devy stores in your Supabase

Every category below is stored in a Postgres schema inside your Supabase project, behind Row-Level Security policies that scope every row to your user.

CategoryWhat it containsWhy it's stored
Identity The email address you signed in with, the provider you used, and a Supabase Auth user ID. To enforce the ALLOWED_EMAIL gate and identify rows you own.
Provider tokens OAuth access tokens, refresh tokens, expiry, and granted scopes for each connected provider, encrypted at rest with the Supabase-managed key. To call the provider APIs on your behalf when polling and when you trigger a Focus session.
Signal rows Per-item metadata pulled from each provider — see the next section for the exact fields. Includes a stable provider ID, a title, a URL, a state flag, timestamps, and a pulse classification. To render your inbox and the pulse charts.
Pulse rollups Hourly counts per source (number of new mentions, number of meetings, etc). No row-level content. To draw the long-range pulse chart without keeping raw signals forever.
Settings Theme, density, alert thresholds, the AI provider you chose and the API key you pasted, the Slack/Calendar IDs to use during Focus. To remember how you like the app configured.
Audit log An append-only record of provider connections, Focus sessions, and settings changes. So you can see exactly what Devy has done with your accounts.

4. What we read from each provider

Devy uses the narrowest scope each provider offers for the features it ships. We never call broader endpoints than we need to, and we never read message bodies, file contents, or diffs.

GitHub

Scope: notifications, read:user. Endpoints: GET /notifications on a 60-second poll. From each notification we keep the repo full-name, the subject title, the type (Issue / PullRequest / Mention), the URL, the unread state, and the updated-at timestamp. We do not call GET /repos/<repo>/issues/<n>, we do not read PR diffs, and we do not store any source code.

Google Calendar

Scope: https://www.googleapis.com/auth/calendar.events, userinfo.email. Endpoints: events.list for the next 24 hours on a 5-minute poll, events.insert only when you start a Focus session. We keep the event ID, the start/end, the title, the attendee count (number, not list), and a flag indicating whether the event has a video conference link. We do not store attendee email addresses, descriptions, or attachments.

Slack

Scope: channels:history, groups:history, users:read, users.profile:write, dnd:write. Endpoints: search.messages for mentions on a 90-second poll. We keep the channel name, the message permalink, the author's display name, the snippet Slack returns in search results (≤120 chars), and the timestamp. We do not call conversations.history on whole channels and do not store full message threads or DMs.

5. What Devy writes back

Polling is read-only. The only writes Devy performs to your providers happen when you click "Start Focus":

  • Google Calendar: events.insert to create a single event titled "Focus" on the calendar you selected, with the duration you chose.
  • Slack: users.profile.set to update your status text and emoji for the duration, plus dnd.setSnooze to enable Do Not Disturb. Both are reverted automatically at the end of the session.

Every Focus session is recorded in the audit log with a timestamp, the duration, and the providers that were touched.

6. AI provider data

The morning briefing is generated by an external AI provider you select (Gemini, Groq, OpenAI, Anthropic, OpenRouter, or any compatible OpenAI-style endpoint). When the briefing runs, your Worker sends the AI provider:

  • the list of signal titles and pulse classifications for the last 24 hours,
  • the timestamps and titles of upcoming Calendar events for the next 12 hours,
  • your local timezone.

The Worker does not send raw provider tokens, your email address, message bodies, or any data that wasn't already visible to you in the inbox. Whatever the AI provider does with that prompt is governed by your contract with them — typically their data-use policy for API traffic. We have no relationship with your AI provider and don't see the prompt or response.

7. Retention

  • Signal rows: deleted by a daily Worker cron once they are 90 days old, or on first sight if the provider has marked them as resolved/read more than 7 days ago.
  • Pulse rollups: kept indefinitely. They contain only counts and timestamps — no per-message content.
  • Audit log: kept for 12 months.
  • Provider tokens: kept until you click "Disconnect" in Settings, or until the provider revokes the grant. Disconnecting deletes the token row and calls the provider's token/revoke endpoint where one exists.
  • AI key: kept until you remove it in Settings.

8. Access & deletion

Because the data lives in your own Supabase, you have complete and direct access:

  • Export: any of the tables under the devy.* schema can be exported as CSV or JSON from the Supabase dashboard or with the standard Postgres tools.
  • Delete a category: truncate devy.signals, truncate devy.audit, etc.
  • Delete everything: drop schema devy cascade from the Supabase SQL editor wipes all Devy-managed data; the Supabase user account itself can be removed from the Auth dashboard.
  • Decommission entirely: delete the Cloudflare Worker, then delete the Supabase project. After that, no Devy-managed data exists anywhere.

Where applicable law (e.g. UK GDPR, EU GDPR, CCPA) gives you rights of access, rectification, erasure, portability or restriction, you can exercise them yourself against your Supabase project — there is no separate request flow because the data is already in your hands.

9. Cookies & local storage

The Devy SPA sets the following items in your browser, all on the deployment's own origin:

  • Session cookie issued by Supabase Auth — HttpOnly, Secure, SameSite=Lax. Required for sign-in.
  • Local-storage keys for the active workspace, theme, density, and the last-viewed inbox tab. These are conveniences and never leave the browser.
  • Session-storage keys for the in-flight OAuth state nonce while a connection flow is open.

Devy sets no analytics, advertising, or third-party tracking cookies.

10. Telemetry

The Devy Worker emits no telemetry to any party other than the providers it integrates with. We do not run analytics, error reporters, or product-usage SDKs in either the Worker or the SPA. If you choose to enable Cloudflare Web Analytics, Sentry, or another tool on your own deployment, that is your decision and is governed by that tool's policy.

11. Security

Devy relies on the security model of the platforms it runs on:

  • OAuth tokens are stored encrypted at rest by Supabase (AES-256, managed key).
  • Row-Level Security is enabled on every table; the Worker uses the user's Supabase JWT, not the service-role key, for read paths.
  • The auth-proxy uses an HMAC-signed state with a 10-minute TTL and a same-origin cookie binding to defeat CSRF and replay.
  • The SPA sets a strict Content-Security-Policy, X-Frame-Options: DENY, and Referrer-Policy: strict-origin-when-cross-origin.

Vulnerability disclosures should follow the process in SECURITY.md — please don't file public issues for security bugs.
