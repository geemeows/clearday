3. The auth-proxy service

Clearday-the-project registers OAuth applications with each integration provider (GitHub, Google, Slack) so that individual self-hosters do not have to register their own. Provider redirect URIs point to a single Cloudflare Worker we operate at https://auth.clearday.geemeows.com ("auth-proxy").

The auth-proxy is stateless. When a provider redirects a user there with an OAuth code and a signed state, the proxy verifies the state's HMAC and TTL (10 minutes), then 302-redirects the request to the user's own backend at <userBackendUrl>/oauth/exchange. The auth-proxy never performs the code → token exchange and never persists tokens, codes, or any data about the request. Your tokens are only ever exchanged on, and stored on, your own infrastructure.

We may inspect, log, rate-limit, or temporarily block traffic to the auth-proxy where necessary to keep it available to other users (for example, in response to abuse or a denial-of-service event). We may change the deployment, rotate the OAuth client secrets, move the proxy to a new domain, or shut it down with reasonable notice. If we shut it down, you can keep using Devy by registering your own OAuth applications with each provider — the proxy is a convenience, not a dependency.

We do not offer the auth-proxy under an SLA. It is a best-effort, free-to-use service for users of the open-source project.

If a provider revokes its OAuth grant, your Worker stops receiving signals from that provider until you reconnect.

6. Single-tenant restriction

Each Devy deployment is intended to be operated by exactly one engineer — its owner. The Worker enforces this with two checks: a Postgres Row-Level Security policy that filters every query by user, and a Worker-side comparison of the authenticated email against an ALLOWED_EMAIL environment variable. Sessions whose email does not match are rejected before any data is read.

You may set ALLOWED_EMAIL to your own address only. You may not run a single deployment that exposes Devy to multiple users; running Devy as a multi-tenant SaaS product would require changes to the codebase and to these Terms that the open-source project does not currently authorise.

7. AI provider keys

The key is stored in your Supabase, read by your Worker, and sent only to the provider you selected.

Your usage of that AI provider is governed by that provider's terms, including any limits on commercial use of free-tier credits. We have no relationship with your AI provider and cannot mediate disputes between you and them.
