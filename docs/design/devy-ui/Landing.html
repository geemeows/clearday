What Devy does

A daily driver for the engineer's attention.

Built around the four things that ruin a maker day: pings you missed, pings you didn't need, meetings you forgot, and the noise of switching between five tabs to figure out what's actually urgent.

01 · Triage

Mentions, meetings & reviews — graded by urgency.

Every signal lands in the same inbox with a pulse score: Now for things that need a reply within the hour, Soon for today, FYI for the timeline. Slack mentions, GitHub review requests, and calendar events with a video link share one queue.

06 · Read-only by default

Devy doesn't archive your inbox, post in your channels, or close your PRs.

Polling is read-only on every provider. The only writes Devy ever performs are the two you start a Focus session with — a Calendar event and a Slack status / DnD — and they revert when the timer hits zero. Every action lands in an audit log inside your Supabase, append-only, that you can SELECT from any time.

How it works

Your data takes the shortest possible path home.

Devy is single-tenant by construction. There's no shared database, no shared backend, no central tenant table. The auth-proxy we operate is a stateless redirect — it never sees a token.

1
Single-tenant by default.

Each deployment binds to one ALLOWED_EMAIL. The Worker rejects any session that doesn't match before a single row is read. Postgres RLS adds a second wall under it.

2
Stateless auth-proxy.

The OAuth code is in the proxy's memory for the millisecond it takes to verify an HMAC and emit a 302 — no logs, no DB, no analytics. Your tokens are exchanged exclusively between your Worker and the provider.

3
Narrowest scope per provider.

GitHub notifications, Calendar events, Slack search.messages. Devy never reads message bodies, file contents, or PR diffs.

4
Bring-your-own AI key.

The morning briefing is sent only to the provider you chose. Your key lives in your Supabase, used by your Worker. We don't broker AI traffic and never see the prompt or response.

Self-host

Yours in five minutes.

A Cloudflare Worker, a Supabase project, two secrets. Devy ships a deploy script that handles the rest — including registering your ALLOWED_EMAIL and seeding the schema with RLS already on.

01
Clone & configure git clone the repo, create a Supabase project, drop in your ALLOWED_EMAIL.
02
Run the deploy script One command pushes the schema, registers the Worker, and prints your deployment URL.
03
Sign in & connect Open the URL, log in via Google or GitHub, click through three OAuth screens — done.
04
Drop in an AI key (optional) Pick a provider, paste a key, get a briefing tomorrow at 09:00.
